# Apple TV 风格 Android TV 启动器设计规格

## 1. 目标

构建一个可替换系统默认桌面的 Android TV 启动器，最低支持 Android 6（API 23），以 Xiaomi TV Box S 的 4K 输出环境作为性能验收设备，并兼顾 1 GB 内存的低端电视盒子。

产品重点是 Apple TV 风格的横向大卡片、六列网格、留白与焦点空间感，而不是复制 Apple 的商标、图标或其他受保护素材。

首版必须满足：

- 可注册为系统 `HOME` 应用并替换默认桌面。
- 默认桌面为空，中央显示“按菜单键添加应用”。
- 设置页可添加应用，并配置每行数量、卡片尺寸和卡片内图标尺寸。
- 默认每行 6 张横向大卡片，不提供大分组或 Top Shelf。
- 非焦点卡片只有极浅投影且不显示应用名称。
- 焦点卡片明显放大、上浮并显示大范围软投影，同时显示应用名称。
- 遥控器长按确认键或鼠标长按卡片时，提供“移动”和“隐藏”操作。
- 刷新率保持当前分辨率并请求同分辨率下最高模式，上限 120 Hz；请求失败时跟随系统当前模式。
- 桌面静止时不持续渲染，不以牺牲内存换取流畅度。

## 2. 非目标

- 不实现影视内容聚合、推荐流、Top Shelf 或大分组。
- 不实现动态航拍视频、实时模糊、粒子或 3D 场景。
- 不使用 Flutter、Compose、WebView、Leanback、Rust 或自研 OpenGL 渲染器。
- 不解压或修改已安装 APK；应用资源统一通过 Android `PackageManager` 读取。
- 不常驻后台 Service，不轮询应用安装状态。

## 3. 技术架构

项目采用原生 Java、Android View 和 XML，`minSdkVersion` 为 23。首版只包含以下主要单元：

1. `LauncherActivity`：声明 `MAIN`、`HOME` 和 `DEFAULT`，承载全屏桌面、按键分发和应用启动。
2. `LauncherGridView`：基于系统 `GridView`，负责六列布局、焦点卡片最后绘制、方向键导航和卡片复用。
3. `AppGridAdapter`：绑定已加入桌面的应用条目，不在焦点移动时重新创建或重新绑定整个网格。
4. `SettingsActivity`：应用添加列表与列数、卡片尺寸、图标尺寸、壁纸配置。
5. `AppRepository`：通过 `PackageManager` 枚举可启动 Activity，读取 Banner、图标、标签和安装状态。
6. `BannerLoader`：在有界后台线程中准备目标尺寸图片，并管理有上限的内存缓存与磁盘派生缓存。
7. `LauncherPreferences`：使用 `SharedPreferences` 和平台内置 `org.json` 保存有序组件列表及显示设置。
8. `DisplayModeController`：选择与当前分辨率相同、刷新率最高且不超过 120 Hz 的显示模式。

不引入依赖注入、数据库、图片框架或单实现接口。组件之间使用直接、窄范围的方法调用。

## 4. 桌面视觉

### 4.1 整体结构

- 全屏沉浸式静态风景壁纸，不显示状态栏、导航栏、时钟或顶部分组。
- 图标区保留 Apple TV 式上方留白，默认从画面中部附近开始排列。
- 默认 6 列；列数改变后根据可用宽度重新计算卡片宽度，卡片保持约 16:9 的横向比例。
- 卡片圆角按实际卡片高度的约 12% 计算，使不同尺寸和输出分辨率保持相同视觉比例。
- 网格允许焦点卡片超出单元边界绘制，且焦点卡片始终位于其他卡片上层。

### 4.2 焦点状态

非焦点卡片：

- 缩放为 `1.0`。
- 使用约 `1dp` 的极浅 elevation，并配合低不透明度环境阴影。
- 不显示应用名称。
- 不改变壁纸亮度，不触发全屏 alpha 动画。

焦点卡片：

- 默认缩放至 `1.12`，向上平移约 `6dp`。
- elevation 提升至约 `24dp`，形成明显的大范围软投影。
- 应用名称在卡片下方淡入，并保持足够空间，不与下一行重叠。
- 圆角轮廓参与 elevation 阴影计算，图片内容裁切到同一圆角。

焦点切换时只动画离开的卡片和进入的卡片。默认时长约 `180ms`，使用快速起步、平滑停止的插值曲线；新方向输入会取消旧动画并从当前视觉状态继续，避免排队和输入延迟。

### 4.3 空桌面

当有序桌面列表为空时，隐藏网格并在画面中央显示“按菜单键添加应用”。收到 `KEYCODE_MENU` 后直接打开设置页。桌面加入第一个应用后，提示消失并将焦点放到第一张卡片。

## 5. 应用图片策略

应用添加列表同时查询：

- `ACTION_MAIN + CATEGORY_LEANBACK_LAUNCHER`
- `ACTION_MAIN + CATEGORY_LAUNCHER`

结果按组件去重，允许电视应用与只有普通 Launcher Activity 的侧载应用被添加。

横向图片读取顺序为：

1. 启动 Activity 声明的 Banner。
2. Application 声明的 Banner。
3. 若均不存在，生成与其他卡片相同尺寸和圆角的横向卡片，将 APK 的原始方形图标等比缩放后居中放置，绝不横向拉伸。

卡片内图标尺寸由设置控制，只改变图标内容，不改变卡片外框。图片在后台按目标卡片像素尺寸生成，缓存键包含组件名、应用更新时间、卡片尺寸和图标尺寸；应用升级或尺寸配置变化时自动失效。

派生图片磁盘缓存只保存已加入桌面的应用，使用标准文件 API 和原子替换；内存缓存设置硬上限，不能按应用总数无限增长。首次未加载完成时显示稳定的中性占位卡片，图片到达后只更新对应条目。

## 6. 交互与设置

### 6.1 桌面操作

- 方向键：在网格中移动焦点。
- 短按确认键或鼠标单击：启动对应 Activity。
- 长按确认键或鼠标长按：显示只有“移动”和“隐藏”的操作菜单。
- 菜单键：从桌面任意状态进入设置。
- 返回键：在移动模式中取消并恢复进入移动模式前的顺序；普通桌面不退出 Launcher。

移动模式中，方向键将当前卡片与目标位置交换并播放短位移动画；确认键提交顺序，返回键取消。隐藏操作立即从桌面列表移除，但不卸载应用；用户可在设置页重新添加。

### 6.2 设置页

设置页使用适合遥控器的原生列表控件，包含：

- 应用管理：显示所有可启动应用及是否已加入桌面，确认键切换添加状态。
- 每行数量：提供合理范围内的整数选择，默认 6。
- 卡片尺寸：使用离散档位或步进值，改变卡片整体尺寸并保持横向比例。
- 图标尺寸：独立控制缺少 Banner 时居中的方形图标大小。
- 壁纸：使用内置默认静态壁纸，并允许通过系统文件选择器选择本地图片。

设置改变后立即持久化；返回桌面时一次性重新计算布局或失效相关图片，不在拖动设置控件的每一帧重建全网格。

## 7. 性能设计

### 7.1 主线程

- 主线程不读取 APK 文件、不解码 Bitmap、不扫描全部包、不写磁盘。
- 焦点移动不调用全网格刷新，不改变 LayoutParams，不触发重新测量。
- `scale`、`translationY`、`elevation` 和名称透明度使用属性动画，由硬件加速渲染路径执行。
- 动画按 `Choreographer` 的实际 VSync 运行，不自行创建固定 60 FPS 的定时器。
- 连续方向输入只保留当前目标状态，不累计动画任务。

### 7.2 图片与内存

- 壁纸最大按 1920x1080、ARGB_8888 解码，在 4K 输出时由 GPU 缩放，避免单张 4K 壁纸占用约 32 MB Java 堆。
- Banner 只按显示所需尺寸生成，不保留 APK 原始大图与派生 Bitmap 两份副本。
- 缓存上限根据设备内存档位设置，但 1 GB 设备采用保守固定上限；收到内存回收回调时主动缩减。
- 卡片离屏后可复用 View，图片缓存淘汰不影响顺序和交互状态。

### 7.3 显示模式

`DisplayModeController` 只考虑与当前物理宽高完全相同的模式，选取最高刷新率且不超过 120 Hz 的模式 ID。若没有更高模式、API/厂商拒绝请求、窗口实际模式未改变或发生异常，则不重复切换，直接跟随系统当前刷新率。

这项策略不降分辨率，不通过忙循环“补帧”，也不假定高性能盒子一定获得厂商 Launcher 同等调度优先级。

## 8. 数据与状态流

启动流程：

1. 读取轻量显示设置和有序组件名列表。
2. 立即显示壁纸、空桌面提示或稳定占位卡片。
3. 后台核对组件是否仍可启动并加载可见卡片图片。
4. 主线程按组件键更新仍然有效的对应卡片。
5. 返回桌面或收到安装、卸载、升级事件时增量核对，不启动常驻扫描。

应用顺序以完整组件名保存，避免同一包多个入口冲突。应用卸载后自动从桌面移除；应用升级保留位置但失效对应图片缓存。

## 9. 错误处理

- 横向 Banner 缺失或加载失败：使用居中方形图标的生成卡片。
- 图标也无法读取：使用统一中性占位符和应用名称，不崩溃。
- 应用启动失败：重新核对组件并显示简短提示；确认已卸载时移除该条目。
- 配置损坏：保留可解析字段，其余恢复默认；无法解析有序列表时回到空桌面。
- 自定义壁纸不可访问或解码失败：回退到内置壁纸。
- 显示模式请求失败：跟随系统模式，不反复请求、不闪屏。
- 图片异步结果到达时条目已复用或删除：按组件键和请求代次丢弃过期结果。

## 10. 验证标准

功能验证至少覆盖：

- Android 6 环境可安装并被系统选择为默认 HOME。
- 首次启动为空桌面，菜单键可进入设置。
- 普通 Launcher 与 Leanback 应用均可添加，顺序可保存、移动、取消移动和隐藏。
- 有 Banner 时使用横向大图；无 Banner 时方形图标等比居中。
- 非焦点无应用名且投影极浅；焦点有名称、大圆角、大投影和连续动画。
- 列数、卡片尺寸、图标尺寸及壁纸修改在重启后保留。
- 应用卸载、升级、无法启动、壁纸失效和配置损坏都有确定降级行为。
- 同分辨率刷新率选择逻辑覆盖 60/90/120 Hz、无匹配模式和请求失败。

性能验证在 Xiaomi TV Box S 的实际输出模式上通过 ADB 完成：

- 预热后连续执行至少 500 次方向焦点移动，使用 `dumpsys gfxinfo ... framestats` 检查帧时间。
- 在 60 Hz 模式下，稳态焦点动画的第 95 百分位帧时间不超过一个 VSync 周期，并将超时帧控制在 1% 以内。
- 若设备和显示器实际提供 120 Hz 同分辨率模式，则同样按 8.33 ms VSync 周期检查；设备没有该模式时不得宣称已验证 120 FPS。
- 焦点移动期间不得发生 Bitmap 解码、包扫描、磁盘访问或整个网格重新绑定。
- 连续移动 1000 次后 Java 堆不持续增长；触发 GC 后图片缓存回到配置上限内。
- 分别记录 Java Heap、Native Heap、Graphics 和总 PSS，避免把 4K Surface 缓冲区误判为普通对象泄漏。
- 冷启动、首次图片生成和缓存预热分别测量，不用预热成绩掩盖首次加载卡顿。

若 Xiaomi TV Box S 仍出现第三方桌面特有掉帧，先用相同 APK、相同 HDMI 模式和相同操作序列收集 `gfxinfo framestats`、`dumpsys SurfaceFlinger`、`dumpsys display` 与 `dumpsys meminfo`，区分应用主线程、GPU 合成、显示模式和厂商调度后再做单一变量修复。
